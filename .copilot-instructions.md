# Copilot Instructions

## Project overview

This repository is a **CodeIgniter 3** application written in **PHP** with a classic MVC structure.

Main goals when generating code:
- preserve compatibility with CodeIgniter 3
- make minimal and safe changes
- generate copy-paste-ready code
- avoid unnecessary refactors
- respect the existing project structure

This project also uses a **local DICOM library** already available in the codebase or server environment.  
When a task involves DICOM files, studies, series, metadata, tags, image extraction, parsing, or conversion, prioritize integrating with the existing **local DICOM library** instead of proposing external services or replacing the current approach.

---

## Mandatory rules

### 1. Framework compatibility
Always keep compatibility with **CodeIgniter 3**.

Use patterns consistent with CI3:
- `CI_Controller`
- `CI_Model`
- `$this->load->model()`
- `$this->load->library()`
- `$this->load->helper()`
- `$this->load->view()`
- `$this->input->post()`
- `$this->input->get()`
- `$this->db` with Query Builder when applicable

Do not propose migration to CodeIgniter 4, Laravel, Symfony, or other frameworks.

---

### 2. Minimal change policy
Prefer the smallest possible change that solves the problem.

Do:
- modify only what is necessary
- preserve method names and public interfaces unless explicitly asked
- keep file structure unchanged
- return focused patches and fragments when requested

Do not:
- rewrite unrelated code
- introduce large architectural changes
- rename existing classes, methods, routes, or views without need

---

### 3. Respect MVC
- **Controllers** handle request flow, validation coordination, and responses
- **Models** handle database access and data retrieval/persistence
- **Views** handle presentation and should contain minimal logic

Avoid putting SQL-heavy logic inside controllers.  
Avoid business logic in views.

---

### 4. Follow the existing stack
Assume this project commonly uses:
- PHP
- MySQL / MariaDB
- jQuery for AJAX
- Bootstrap for UI
- HTML/CSS/JS in a traditional server-rendered setup

Prefer:
- jQuery for AJAX if frontend examples already use it
- Bootstrap-compatible markup if UI code is needed
- existing helpers/libraries before adding new abstractions

---

### 5. Security and robustness
Whenever applicable:
- validate inputs
- sanitize user data appropriately
- avoid SQL injection
- prefer Query Builder over unsafe raw SQL
- escape output in views when needed
- preserve CSRF/XSS protections already in place
- handle null/empty/error cases explicitly

---

## DICOM-specific instructions

This project uses a **local DICOM library**.

When working on DICOM-related tasks, Copilot must follow these rules:

1. **Do not replace the local DICOM library**
   - Do not propose cloud APIs or external DICOM services by default
   - Do not introduce a different DICOM package unless explicitly requested

2. **Integrate with the existing local library**
   - Reuse the current library loading mechanism
   - If the project uses `application/libraries`, keep using that pattern
   - If there is already a wrapper/helper for DICOM operations, extend it instead of duplicating logic

3. **Encapsulate DICOM operations**
   - Prefer calling DICOM logic from a library, helper, or model/service-like existing component
   - Keep controllers thin
   - Do not scatter DICOM parsing logic across views/controllers

4. **Be explicit about assumptions**
   If the exact DICOM library API is unknown, state assumptions clearly such as:
   - assumed class name
   - assumed initialization path
   - assumed methods for reading metadata/tags/images

5. **Typical DICOM tasks**
   Copilot may help with:
   - reading DICOM headers
   - extracting metadata
   - accessing patient/study/series information
   - rendering or exporting previews if the existing library supports it
   - mapping DICOM data to database fields
   - uploading and processing `.dcm` files
   - validating file presence and error handling

6. **Error handling**
   DICOM-related code should handle:
   - missing file
   - unreadable file
   - unsupported transfer syntax if relevant
   - missing tags
   - library load/init failure

7. **Performance**
   For DICOM operations:
   - avoid reparsing the same file multiple times
   - avoid loading full datasets in controllers if a library can encapsulate that
   - cache or persist parsed metadata only if the existing project pattern supports it
