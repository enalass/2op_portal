GUÍA DE ESTRUCTURA DE BASE DE DATOS PARA MODIFICAR FORMULARIOS Y MODELOS

Objetivo
Este proyecto no usa relaciones declaradas mediante FOREIGN KEY en la base de datos. Las relaciones entre tablas se deducen por convención de nombres y se resuelven en el código.

El objetivo de esta guía es que cualquier agente (por ejemplo Copilot) pueda interpretar correctamente la estructura de tablas, detectar claves primarias y foráneas, y aplicar cambios coherentes en formularios, modelos, consultas y lógica asociada.

----------------------------------------
1. CONVENCIÓN GENERAL DE NOMBRES
----------------------------------------

Cada tabla tiene un prefijo de 3 letras que identifica su entidad.
Ese prefijo aparece también al inicio del nombre de sus columnas.

Ejemplo:
- Tabla: t_sol_solicitudes
- Prefijo de la tabla: SOL

Columnas de ejemplo de esa tabla:
- SOL_CO_ID
- ESO_CO_ID
- FSO_CO_ID
- SOL_DS_NOMBRE
- SOL_DT_CREATE
- SOL_BL_ENABLE
- SOL_BL_DELETE

Regla principal:
- Si una columna termina en _CO_ID y el prefijo de esa columna coincide con el prefijo de la tabla, entonces normalmente es la clave primaria de esa tabla.
- Si una columna termina en _CO_ID y su prefijo NO coincide con el prefijo de la tabla, entonces debe interpretarse como una clave foránea lógica hacia otra tabla.

----------------------------------------
2. TIPOS DE TABLAS SEGÚN EL PREFIJO DEL NOMBRE
----------------------------------------

Convención habitual:
- m_*** : tablas maestras, catálogos o auxiliares
- t_*** : tablas transaccionales o principales

Ejemplos:
- m_eso_estadosolicitud
- t_sol_solicitudes

Esto ayuda a interpretar el rol de cada tabla, aunque la lógica principal de relación depende del prefijo de campos, no del prefijo m_ o t_.

----------------------------------------
3. REGLA PARA IDENTIFICAR LA CLAVE PRIMARIA
----------------------------------------

La clave primaria de una tabla suele ser el campo:
[PREFIJO_TABLA]_CO_ID

Ejemplo:
- Tabla: t_sol_solicitudes
- Prefijo tabla: SOL
- Clave primaria: SOL_CO_ID

Otro ejemplo:
- Tabla: m_eso_estadosolicitud
- Prefijo tabla: ESO
- Clave primaria esperada: ESO_CO_ID

----------------------------------------
4. REGLA PARA IDENTIFICAR CLAVES FORÁNEAS LÓGICAS
----------------------------------------

En una tabla, cualquier campo con formato:
XXX_CO_ID
que NO empiece por el prefijo de esa tabla, debe tratarse como clave foránea lógica.

Ejemplo:
Tabla: t_sol_solicitudes
Prefijo tabla: SOL

Campos:
- SOL_CO_ID      -> clave primaria
- ESO_CO_ID      -> clave foránea lógica hacia la tabla con prefijo ESO
- FSO_CO_ID      -> clave foránea lógica hacia la tabla con prefijo FSO

Importante:
Estas relaciones pueden no existir físicamente como FOREIGN KEY en MySQL.
Aun así, deben respetarse al modificar:
- modelos
- formularios
- validaciones
- consultas SQL
- joins
- listas desplegables
- lógica de guardado/edición

----------------------------------------
5. CÓMO DEDUCIR LA TABLA DE DESTINO DE UNA CLAVE FORÁNEA
----------------------------------------

Para una columna XXX_CO_ID:
- XXX es el prefijo de la tabla relacionada
- la tabla destino será la que tenga como clave primaria XXX_CO_ID

Ejemplo:
- ESO_CO_ID debe apuntar a una tabla cuyo prefijo sea ESO
- normalmente esa tabla será algo como: m_eso_estadosolicitud
- y su clave primaria será: ESO_CO_ID

Otro ejemplo:
- FSO_CO_ID debe apuntar a una tabla cuyo prefijo sea FSO
- la tabla exacta se deduce buscando la tabla cuyo identificador principal sea FSO_CO_ID

Si existen varias tablas con el mismo prefijo, debe usarse la que funcionalmente corresponda al contexto del módulo.

----------------------------------------
6. INTERPRETACIÓN DE SUFIJOS DE COLUMNAS
----------------------------------------

Aunque puede haber variaciones, estos sufijos suelen indicar el tipo de dato o propósito:

- _CO_ID       -> identificador numérico / clave
- _DS_         -> descripción o texto
- _DT_         -> fecha/hora
- _BL_         -> booleano / bandera lógica

Ejemplo real:
- SOL_CO_ID         -> identificador de la solicitud
- SOL_DS_NOMBRE     -> nombre o descripción de la solicitud
- SOL_DT_CREATE     -> fecha de creación
- SOL_BL_ENABLE     -> activo/inactivo
- SOL_BL_DELETE     -> borrado lógico

Estas convenciones deben mantenerse al crear nuevos campos o modificar modelos existentes.

----------------------------------------
7. EJEMPLO EXPLICADO
----------------------------------------

SQL de ejemplo:

CREATE TABLE `t_sol_solicitudes` (
  `SOL_CO_ID` int(11) NOT NULL,
  `ESO_CO_ID` int(11) NOT NULL,
  `FSO_CO_ID` int(11) NOT NULL,
  `SOL_DS_NOMBRE` varchar(150) COLLATE utf8_unicode_ci NOT NULL,
  `SOL_DT_CREATE` timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `SOL_BL_ENABLE` tinyint(4) DEFAULT '1',
  `SOL_BL_DELETE` tinyint(4) NOT NULL DEFAULT '0'
) ENGINE=InnoDB DEFAULT CHARSET=utf8 COLLATE=utf8_unicode_ci;

Interpretación para el agente:

- Tabla: t_sol_solicitudes
- Prefijo de entidad: SOL
- Clave primaria: SOL_CO_ID

Campos:
- SOL_CO_ID: ID principal de la solicitud
- ESO_CO_ID: relación lógica con la entidad ESO
- FSO_CO_ID: relación lógica con la entidad FSO
- SOL_DS_NOMBRE: nombre de la solicitud
- SOL_DT_CREATE: fecha de creación
- SOL_BL_ENABLE: indicador de activo
- SOL_BL_DELETE: indicador de borrado lógico

Relaciones deducidas:
- t_sol_solicitudes.ESO_CO_ID -> tabla de prefijo ESO, PK ESO_CO_ID
- t_sol_solicitudes.FSO_CO_ID -> tabla de prefijo FSO, PK FSO_CO_ID

Notas funcionales:
- SOL_BL_DELETE = 0 significa registro no borrado
- SOL_BL_DELETE = 1 significa borrado lógico
- SOL_BL_ENABLE = 1 significa activo
- SOL_BL_ENABLE = 0 significa inactivo

----------------------------------------
8. CÓMO DEBE ACTUAR EL AGENTE AL HACER CAMBIOS
----------------------------------------

Cuando se solicite modificar formularios, modelos o lógica, el agente debe seguir estas reglas:

1. Identificar el prefijo de la tabla principal.
2. Detectar su clave primaria como [PREFIJO]_CO_ID.
3. Detectar claves foráneas lógicas como todos los campos XXX_CO_ID cuyo prefijo XXX no coincida con el de la tabla actual.
4. Buscar la tabla de referencia por el prefijo XXX.
5. Mantener coherencia entre:
   - nombre de campos
   - formularios
   - validaciones
   - consultas SQL
   - joins manuales
   - labels y combos
   - mapeos del modelo
6. Respetar el borrado lógico y los flags de activación cuando existan campos tipo _BL_DELETE o _BL_ENABLE.
7. No asumir que existen restricciones FOREIGN KEY en la base de datos.
8. No renombrar campos ni romper la convención de prefijos.
9. Si se crea una consulta, construir manualmente los JOIN usando las claves deducidas por convención.
10. Si se crea un formulario de edición o alta, usar las tablas maestras relacionadas para poblar selects cuando exista un XXX_CO_ID ajeno al prefijo principal.

----------------------------------------
9. REGLAS PARA GENERAR CONSULTAS O JOINS
----------------------------------------

Como las relaciones están en código, los JOIN deben inferirse.

Ejemplo conceptual:
- tabla principal: t_sol_solicitudes SOL
- campo relacionado: ESO_CO_ID
- tabla relacionada esperada: m_eso_estadosolicitud ESO

Join esperado:
- SOL.ESO_CO_ID = ESO.ESO_CO_ID

Es decir, el lado izquierdo usa el campo foráneo de la tabla principal y el lado derecho la PK de la tabla relacionada.

----------------------------------------
10. REGLAS PARA FORMULARIOS
----------------------------------------

Si un formulario trabaja sobre una tabla principal:
- los campos del propio prefijo son campos nativos de la entidad
- los campos XXX_CO_ID con prefijo distinto son relaciones y normalmente deben mostrarse como select, autocomplete o buscador

Ejemplo en t_sol_solicitudes:
- SOL_DS_NOMBRE -> input de texto
- ESO_CO_ID -> select de estado
- FSO_CO_ID -> select o relación con catálogo FSO

Además:
- no mostrar normalmente la PK en formularios salvo que haga falta para edición interna
- respetar valores por defecto
- respetar flags de activo/borrado si forman parte de la lógica del módulo

----------------------------------------
11. REGLAS PARA MODELOS O ENTIDADES
----------------------------------------

En modelos, cada tabla debe mapear:
- su PK principal
- sus campos simples
- sus relaciones lógicas deducidas por prefijo

Aunque no haya ORM con relaciones explícitas, el agente debe interpretar que:
- XXX_CO_ID es una referencia a otra entidad
- puede ser necesario resolver el valor descriptivo mediante consulta adicional o JOIN manual

- ESO_CO_ID -> tabla con PK ESO_CO_ID

----------------------------------------
13. INSTRUCCIÓN DIRECTA PARA COPILOT / AGENTE
----------------------------------------

A partir de ahora, interpreta la base de datos con estas reglas:

- La clave primaria de cada tabla es el campo [PREFIJO_TABLA]_CO_ID.
- Todo campo XXX_CO_ID cuyo prefijo no coincida con el de su tabla debe tratarse como una clave foránea lógica.
- Las relaciones no están definidas con FOREIGN KEY en MySQL, pero sí existen funcionalmente en la aplicación.
- Los JOIN, validaciones, combos de formularios y modelos deben construirse usando esta convención.
- Debes respetar nombres de columnas, prefijos y lógica de borrado/activación.
- No debes asumir relaciones automáticas del motor de base de datos; hay que deducirlas por nomenclatura.

----------------------------------------
14. RECOMENDACIÓN DE ENTRADA PARA EL AGENTE
----------------------------------------

Cuando se le pase una tabla al agente, conviene enviarla en este formato:

1. SQL CREATE TABLE
2. Explicación del prefijo de la tabla
3. Identificación de la PK
4. Lista de campos que son FK lógicas
5. Nombre de tablas relacionadas si se conocen
6. Reglas funcionales adicionales del módulo

Ejemplo breve:

Tabla: t_sol_solicitudes
Prefijo: SOL
PK: SOL_CO_ID
FK lógicas:
- ESO_CO_ID -> m_eso_estadosolicitud
- FSO_CO_ID -> tabla con prefijo FSO

Reglas:
- filtrar por SOL_BL_DELETE = 0
- usar SOL_BL_ENABLE para estado activo

----------------------------------------
15. CONCLUSIÓN
----------------------------------------

La base de datos debe interpretarse por convención de prefijos, no por restricciones físicas.
El punto clave es:
- PK = campo [prefijo tabla]_CO_ID
- FK lógica = cualquier XXX_CO_ID cuyo prefijo no coincida con el de la tabla actual

Con esta regla, el agente puede entender cómo modificar formularios, modelos, consultas y relaciones de negocio de forma consistente.

--------------------------------------------------
16.9. REGLAS PARA EL AGENTE AL MODIFICAR FORMULARIOS Y MODELOS
--------------------------------------------------

El agente debe respetar las siguientes reglas al generar o modificar código:

1. No asumir que existen FOREIGN KEY físicas en la base de datos.
2. Resolver relaciones por convención de nombres:
   - Si un campo XXX_CO_ID no coincide con el prefijo de la tabla, debe tratarse como relación lógica.
3. Usar siempre los nombres reales de columnas definidos en SQL.
4. Respetar tipos de datos exactos:
   - int, bigint, varchar, text, datetime, timestamp, tinyint
5. Respetar los flags lógicos existentes:
   - BL_ENABLE / BL_DELETE
   - BL_ACEPTADO / BL_DESHABILITADO / BL_ELIMINADO
6. No eliminar ni renombrar columnas sin necesidad explícita.
7. En formularios:
   - Los campos de relación lógica deben poder resolverse mediante select, búsqueda o carga de catálogo cuando aplique.
8. En modelos:
   - Deben contemplarse las relaciones lógicas aunque no existan restricciones SQL.
9. En listados y consultas:
   - Puede ser necesario hacer joins manuales por convención de claves.
10. Debe respetarse especialmente el caso de t_log_accesos, cuya PK real es LOC_CO_ID aunque la tabla tenga prefijo funcional LOG.
